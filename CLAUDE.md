# Правила проекта «Войны заправок»

1. Перед изменением кода изучи связанные модели, сервисы, схемы и тесты.
2. Не помещай бизнес-логику в FastAPI-роуты.
3. Все денежные изменения проводи через FinancialTransaction.
4. Все игровые сущности должны быть привязаны к game_id.
5. Сервер является источником истины.
6. Не доверяй значениям денег, топлива, цены и владельца от frontend.
7. Используй async SQLAlchemy.
8. Для конкурентных операций используй транзакции и блокировки.
9. После изменения схемы создавай Alembic-миграцию.
10. Для новой бизнес-логики добавляй тесты.
11. Не переписывай существующие модули без необходимости.
12. Не создавай большие универсальные service-классы.
13. Разделяй: API; schemas; services; repositories; simulation; websocket.
14. Все коэффициенты игры должны быть настраиваемыми (в settings_json игры, не в коде).
15. Маршрут хранится как набор точек и временных параметров.
16. Не отправляй позицию каждого транспорта каждый кадр.
17. WebSocket-события отправляй после успешной транзакции БД.
18. Не добавляй функции, которых нет в текущем этапе ТЗ.
19. После выполнения задачи запускай: formatter, linter, type checker, tests.
20. В отчете перечисляй: измененные файлы; миграции; API; тесты; команды запуска; известные ограничения.

## Полный технический план

Полное ТЗ — в [TECHNICAL_SPEC.md](./TECHNICAL_SPEC.md). Разработка ведется строго по этапам
(раздел 36): нельзя переходить к следующему этапу, пока текущий не реализован, не проверен
тестами/линтером/тайпчекером и не соответствует критериям готовности.

## Команды проверки

Backend (`cd backend && source .venv/bin/activate`):

```bash
ruff check .
ruff format --check .
mypy app
pytest
```

Frontend (`cd frontend`):

```bash
npm run lint
npm run typecheck
npm run build
```
